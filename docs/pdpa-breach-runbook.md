# คู่มือรับมือเหตุละเมิดข้อมูลส่วนบุคคล (Data Breach Runbook)

> ตาม PDPA **มาตรา 37(4)** — ผู้ควบคุมข้อมูลต้องแจ้งเหตุละเมิดต่อสำนักงานคณะกรรมการคุ้มครองข้อมูลส่วนบุคคล (สคส./PDPC)
> **ภายใน 72 ชั่วโมง** นับแต่ทราบเหตุ (เว้นแต่ไม่มีความเสี่ยงต่อสิทธิเสรีภาพของบุคคล)
> และหากเสี่ยงสูง ต้องแจ้ง **เจ้าของข้อมูล** ด้วย

**หมายเหตุ:** แบบร่างเพื่อเริ่มต้น ควรให้ที่ปรึกษากฎหมายทบทวน และซ้อมทีมงานอย่างน้อยปีละครั้ง

## 1. ตรวจพบ & แจ้งเหตุภายใน (นาทีที่ 0)
- ใครก็ตามที่พบสิ่งผิดปกติ (ข้อมูลหลุด, เข้าถึงผิดสิทธิ์, ระบบถูกบุกรุก, ไฟล์รั่ว) → แจ้ง **Incident Lead** ทันที
- เริ่ม **บันทึกเวลา (timeline)** ทุกขั้นตอนตั้งแต่วินาทีแรก (ใช้เป็นหลักฐาน)
- **เวลาที่ "ทราบเหตุ" = จุดเริ่มนับ 72 ชม.** → บันทึกให้ชัด

## 2. จำกัดความเสียหาย (Contain) — เร็วที่สุด
- ตัดการเข้าถึงที่ถูกบุกรุก: เพิกถอน/หมุน key ที่รั่ว (`SUPABASE_SERVICE_ROLE_KEY`, `CRON_SECRET`, LINE/SMS tokens)
- ระงับบัญชี/เซสชันที่น่าสงสัย, บังคับ logout, รีเซ็ตรหัสผ่านถ้าจำเป็น
- ถ้าเป็นช่องโหว่โค้ด → deploy patch/ปิดฟีเจอร์ชั่วคราว
- เก็บหลักฐาน (log, snapshot) ก่อนแก้ อย่าทำลายร่องรอย

## 3. ประเมิน (Assess)
ตอบให้ได้ว่า:
- ข้อมูล**ประเภทใด** รั่ว (มีเลขบัตร ปชช./สลิป/ข้อมูลอ่อนไหวหรือไม่)
- **จำนวน**เจ้าของข้อมูลโดยประมาณ
- **สาเหตุ** (แฮก / ตั้งค่าผิด / คนใน / บั๊ก)
- **ความเสี่ยงต่อบุคคล** สูง/ต่ำ (เช่น เสี่ยงถูกสวมรอย, ฉ้อโกง)
- ควบคุมได้แล้วหรือยัง

## 4. แจ้ง สคส. (PDPC) ภายใน 72 ชม.
- หากมีความเสี่ยงต่อสิทธิเสรีภาพของบุคคล → **แจ้ง สคส.** ผ่านช่องทางที่ สคส. กำหนด (เว็บไซต์ PDPC)
- เนื้อหาที่ต้องมี: ลักษณะเหตุ, ประเภท+จำนวนข้อมูลโดยประมาณ, ผลกระทบที่อาจเกิด, มาตรการที่ดำเนินการ/จะดำเนินการ, ช่องทางติดต่อ
- ถ้าเกิน 72 ชม. ต้องระบุเหตุผลความล่าช้า

## 5. แจ้งเจ้าของข้อมูล (ถ้าเสี่ยงสูง)
- แจ้งผู้ที่ได้รับผลกระทบ "โดยไม่ชักช้า" พร้อมคำแนะนำป้องกันตัว (เช่น เปลี่ยนรหัสผ่าน, ระวังมิจฉาชีพ)
- ช่องทาง: อีเมล / SMS / LINE / ประกาศในระบบ

## 6. กรณีข้อมูล "ผู้เช่า" (บริษัทเป็นผู้ประมวลผล)
- ถ้าเหตุเกิดกับข้อมูลผู้เช่า → บริษัท (ผู้ประมวลผล) ต้อง **แจ้งเจ้าของหอ (ผู้ควบคุมข้อมูล) โดยไม่ชักช้า** เพื่อให้เจ้าของหอแจ้ง สคส. ได้ทัน (ตาม DPA ข้อ 9)

## 7. ฟื้นฟู & ป้องกันซ้ำ (Post-incident)
- ปิดช่องโหว่ถาวร, review สิทธิ์/ความปลอดภัย
- เขียน **post-mortem**: อะไรเกิดขึ้น, ทำไม, แก้อย่างไร, ป้องกันซ้ำอย่างไร
- อัปเดต RoPA/มาตรการความปลอดภัยถ้าจำเป็น
